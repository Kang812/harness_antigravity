# Agent Team Design Patterns (Antigravity 버전)

## 실행 모드: 에이전트 팀 vs 서브 에이전트

두 가지 실행 모드의 핵심 차이를 이해하고 적합한 모드를 선택한다.

### 에이전트 팀 (Agent Teams) — 기본 모드

팀 리더(오케스트레이터)가 `define_subagent`로 에이전트들을 정의하고, `invoke_subagent` 호출 시 `Workspace: "share"` 옵션을 활성화하여 팀을 소환한다. 팀원들은 동일한 작업 공간을 공유하며, `send_message`로 직접 통신하고, 공유 작업 파일(`_workspace/tasks.json`)을 활용해 자체 조율한다.

```
[리더/오케스트레이터] ←→ [팀원A] ←→ [팀원B]
   ↕                     ↕          ↕
   └───────── 공유 작업 공간 (share) ──┘
```

**핵심 도구:**
- `define_subagent`: 팀원 에이전트의 역할과 권한 정의
- `invoke_subagent` (with `Workspace: "share"`): 다중 서브에이전트 스폰 및 공동 작업 공간 바인딩
- `send_message({Recipient: id, Message: msg})`: 특정 팀원에게 메시지 전송 (Recipient는 대화 ID)
- `_workspace/tasks.json`: 공유 작업 목록 관리 파일

**특징:**
- 팀원끼리 직접 대화, 검증 및 보완 가능
- 리더를 거치지 않고 팀원 간 상호 참조 가능
- 공유 공간 내 파일 입출력을 통한 대용량 데이터 전달 용이
- 유기적이고 협력적인 다단계 작업 최적화

**제약:**
- 플랫폼 차원의 실시간 중앙 동기화가 아니므로 파일이나 메시지 프로토콜을 명확히 명시해야 함
- 관리 오버헤드가 단일 에이전트보다 큼

---

### 서브 에이전트 (Sub-agents) — 경량 모드

메인 에이전트가 `invoke_subagent` 도구로 특정 작업을 수행할 서브 에이전트를 독립적으로 호출한다. 각 서브 에이전트는 독립된 임시 워크스페이스(`Workspace: "branch"`)에서 동작하며, 최종 결과만 호출자(메인)에게 반환된다.

```
[메인] → invoke_subagent(branch) → [서브A] → 결과 반환
```

**핵심 도구:**
- `invoke_subagent` (with `Workspace: "branch"` or `"inherit"`): 독립적인 하위 태스크 실행

**특징:**
- 가볍고 빠르며, 컨텍스트가 격리되어 안전함
- 토큰 비용이 효율적
- 병렬 독립 작업 처리에 최적화

**제약:**
- 서브 에이전트 상호 간 직접 통신 불가
- 메인이 결과 취합 및 전체 조율을 단독 처리해야 함

## 에이전트 팀 아키텍처 유형

### 1. 파이프라인 (Pipeline)
순차적 작업 흐름. 이전 에이전트의 출력이 다음 에이전트의 입력.

```
[분석] ─(파일 저장)─→ [설계] ─(파일 저장)─→ [구현]
```

* **팀 모드 적합성**: 순차 의존성이 강해 병렬 통신 이점은 적으나, `Workspace: "share"`를 통해 중간 산출물 파일을 매끄럽게 인계하는 방식으로 구현 가능.

### 2. 팬아웃/팬인 (Fan-out/Fan-in)
병렬 처리 후 결과 통합. 독립적 작업을 동시 수행.

```
         ┌→ [전문가A: official] ──┐
[분배] ─┼→ [전문가B: media] ─────┼→ [통합: 리더]
         └→ [전문가C: community] ─┘
```

* **팀 모드 적합성**: 매우 강력함. `invoke_subagent`로 N개의 서브에이전트를 동시 스폰하고, 공유 공간(`share`)에 각 영역의 조사 결과를 독립 파일로 저장하게 한 뒤, 오케스트레이터가 이를 취합하여 통합.

### 3. 전문가 풀 (Expert Pool)
상황에 따라 적절한 전문가를 선택 호출.

```
[라우터] → { invoke_subagent(A) | invoke_subagent(B) }
```

* **팀 모드 적합성**: 서브 에이전트(경량 모드)가 적합. 필요할 때만 단일 에이전트를 호출하므로 상시 팀이 불필요.

### 4. 생성-검증 (Producer-Reviewer)
생성 에이전트와 검증 에이전트가 쌍으로 동작.

```
[생성자] ──(파일 생성)──→ [검증자] ──(send_message: 피드백)──→ [생성자: 수정]
```

* **팀 모드 적합성**: 팀 모드가 최적. `Workspace: "share"`를 통해 코드를 작성하고, `send_message`를 활용해 "X파일 Y라인의 버그 수정 요망" 같은 실시간 피드백 루프 작동.

## 에이전트 정의 구조

모든 에이전트는 `프로젝트/.agents/{name}.md` 파일에 정의한다.

```markdown
---
name: agent-name
description: "역할 및 트리거 정보"
---

# Agent Name

## 핵심 역할
1. ...

## 작업 원칙
- ...

## 입력/출력 프로토콜
- 입력: `_workspace/` 하위 특정 파일
- 출력: `_workspace/` 하위 결과물 파일

## 팀 통신 프로토콜 (에이전트 팀 모드 시)
- 메시지 수신/발신 제약 및 대상 명시

## 에러 핸들링
- 실패 시 재시도 또는 대체 산출물 전략
```
